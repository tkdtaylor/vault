## Verification ladder

A task earns ­ččí at levels 1ÔÇô4 and Ôťů only at level 5 or 6. The `Verified by` column records which level the row reached.

| Level | Evidence | Status this earns |
|-------|----------|-------------------|
| 1 | Code merged | ­ččí |
| 2 | Unit tests pass (paste verbatim final line of `make check`) | ­ččí |
| 3 | `make fitness` passes (verbatim closing line) | ­ččí |
| 4 | CI passes (`gh run watch <id> --exit-status` Ôćĺ success) | ­ččí |
| 5 | **Validation harness** exercises the live runtime path end-to-end ÔÇö paste the command and the final assertion line | Ôťů |
| 6 | **Operator-observed** ÔÇö operator (or executor via `cargo run` / `npm start` / etc.) saw the targeted behaviour in stdout / logs / UI | Ôťů |

If the task targets runtime-observable behaviour (logging, CLI args, TUI, server endpoints, file outputs, side effects), level 5 or 6 is **required** before flipping to Ôťů. If the task only adds an internal helper covered by unit tests, level 2 may be sufficient ÔÇö but in that case the row's `Verified by` should explicitly say "unit-test-only; no runtime surface" so future readers don't mistake silence for verification.

## Rule

**The task-executor commits at ­ččí by default.** Only the main session (after spec-verifier APPROVE + the appropriate level-5/6 evidence) updates the row to Ôťů, in a separate commit titled `verify: confirm task NNN ÔÇö <level-5/6 evidence>`. This keeps the verification step visible in git history and prevents "merged Ôëá done" drift.
